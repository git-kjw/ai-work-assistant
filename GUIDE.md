# 시작 가이드 & 장기 운영 전략

AI 비서를 처음 시작하는 방법과, 장기적으로 메모리를 건강하게 유지하는 전략.

---

## 시작하기: 첫 2주 로드맵

### 이번 주 (세팅)

1. GitHub에서 프로젝트를 clone하고 Basic Memory vault를 연결한다.
2. 곧 작업할 1~2개 프로젝트의 카드를 만든다.

```
> payment-api 프로젝트 카드 만들어줘, 내가 정보 불러줄게
```

프로젝트 카드가 먼저인 이유: 도메인 분석 노트가 프로젝트 카드와 wikilink로 연결되어야
나중에 복합 질문("이 프로젝트 관련해서 알고 있는 거 전부 보여줘")이 동작한다.

### 다음 주 월요일 (첫 실전)

1. 아침에 할일을 등록한다.

```
> payment-api에서 결제 재시도 로직에 설정 외부화 기능 추가해야 돼. 이번 주까지.
```

2. 기능 추가할 프로젝트의 도메인을 분석한다. 해당 기능의 비즈니스 흐름, 상태 전이,
   DB 스키마, 프로시저, 로직 소재지를 정리한다.

```
> 결제 도메인 전체 흐름 분석해줘. 상태 전이, DB, 프로시저 포함해서.
> 환불 로직 수정해야 하는데 어디를 고쳐야 해?
```

3. 작업하면서 태스크 상태를 업데이트한다.

```
> 재시도 로직 설정 외부화 끝났어
```

### 첫째 주 금요일 (성공 경험)

```
> 이번 주 주간보고 초안 만들어줘
```

이번 주에 쌓인 태스크, 도메인 분석, 미팅 노트를 조합해서 AI가 초안을 만들어준다.
이 순간 "아, 이게 되네"라는 느낌이 와야 시스템이 지속된다.

### 주의: 완벽하게 시작하려고 하지 말 것

프로젝트 카드 10개를 다 채우고, 도메인 분석을 다 하고, 반복업무를 다 등록하고 나서
시작하겠다고 하면 시작 자체를 못 한다.
이번 주에 카드 1~2개, 다음 주에 도메인 분석 1~2개, 태스크 몇 개.
**쓰면서 채우는 게 맞다.** 3개월 후에 뒤돌아보면 자연스럽게 쌓여있을 것이다.

## 일상 리듬

| 주기 | 하는 일 | 소요 시간 |
|------|--------|-----------|
| 매일 아침 | "오늘 할일 요약해줘" | 2분 |
| 업무 중 수시 | 태스크 등록/완료, 미팅 기록, 아이디어 저장 | 대화하면서 자연스럽게 |
| 코드 수정 시 | 해당 도메인 분석 요청 → memory 저장 | 작업의 일부 |
| 매주 금요일 | "주간보고 초안 만들어줘" | 5분 (검토 포함) |

매일 추가로 들이는 시간은 사실상 거의 없다.
도메인 분석은 어차피 코드 수정할 때 하는 것이고, 태스크 등록은 대화 한마디면 된다.

## 장기 메모리 관리

### 메모리의 세 단계

**1단계 — 축적기 (1~2개월)**
노트가 빠르게 늘어나는 시기. 프로젝트 카드, 도메인 분석, 태스크가 쌓이면서
AI 비서가 점점 똑똑해지는 느낌을 받는다. 이 시기는 즐겁다.

**2단계 — 혼란기 (3~6개월)**
여기가 고비다. 노트가 100개를 넘으면 문제가 시작된다.
비슷한 내용의 노트가 중복으로 생기고, 오래된 도메인 분석이 현재 코드와 달라지고,
완료된 태스크와 지난 미팅이 검색에 섞여 나온다.
대부분의 개인 지식 관리 시스템이 이 단계에서 무너진다.

**3단계 — 안정기 또는 폐기 (6개월~)**
2단계를 넘기면 "어떤 노트가 가치 있고 어떤 건 정리해야 하는지" 감이 잡히면서
안정된다. 못 넘기면 시스템을 안 쓰게 된다.

### 2단계를 넘기는 전략

**월 1회 "메모리 정리의 날"을 정한다.**
매월 첫째 주 월요일 30분을 투자한다.

```
> 오래된 도메인 분석 노트 있어? last_reviewed 30일 넘은 거.
> 완료된 태스크 정리해줘
> 지난 미팅 아카이브해줘
```

이 세 마디면 된다. 핵심은 **정기적으로** 한다는 것이고, 한 달에 30분이면 충분하다.

**도메인 분석 노트는 "전부 최신으로 유지"하려고 하지 않는다.**
모든 도메인 분석을 최신으로 유지하는 건 불가능하다.

- 비즈니스 흐름, 상태 전이: 비교적 안정적. 정책이 바뀔 때만 갱신.
- DB 스키마: 테이블 추가/변경 시 갱신.
- 프로시저: 변경 빈도 낮음. 수정할 때 갱신.
- 로직 소재지: 리팩토링 시 갱신. 평소에는 안정적.

도메인 노트가 6개월째 안 바뀌었어도 해당 도메인을 안 건드렸으면 문제가 아니다.
문제는 **건드리는데 옛날 분석을 그대로 믿는 것**이고,
그래서 CLAUDE.md에 `last_reviewed` 30일 경고 규칙을 넣어둔 것이다.

**노트 수가 많아지는 것 자체는 문제가 아니다.**
Basic Memory의 시멘틱 검색이 있으니까 300개든 500개든 검색은 된다.
문제는 **틀린 노트**다. 양보다 정확도를 관리한다.

### 정기 점검 주기

| 주기 | 하는 일 | 소요 시간 |
|------|--------|-----------|
| 매월 1회 | stale 노트 확인, 완료 태스크/미팅 아카이브 | 30분 |
| 분기 1회 | 도메인 분석 노트 갱신, 의존관계 맵 업데이트 | 1시간 |

## 노트 종류별 장기 가치

6개월 후에 돌아보면, 종류별로 가치가 다르다.

### 가치가 계속 높은 것

- **프로젝트 카드**: 잘 안 바뀌면서 계속 참조된다.
- **도메인 분석 (비즈니스 흐름, 상태 전이)**: 비즈니스 규칙은 코드보다 천천히 바뀐다. 한번 쓰면 오래 간다.
- **도메인 분석 (로직 소재지)**: "이거 Java야 프로시저야?"에 즉시 답할 수 있다.
- **서비스 의존관계 맵**: "이거 수정하면 어디에 영향가?"에 매번 쓰인다.
- **의사결정 기록**: 왜 그렇게 결정했는지는 시간이 갈수록 잊혀지니까.
- **장애 분석 기록**: "작년에 비슷한 장애 있었어?"에 답할 수 있다.

### 가치가 점점 떨어지는 것

- **완료된 태스크**: 아카이브로 보내고 필요할 때만 찾는다.
- **지난 미팅 노트**: 후속 조치가 태스크로 빠지면 미팅 노트 자체는 가치가 낮아진다.

### 관리 비용 대비 가치를 따져야 하는 것

- **도메인 분석 (DB 스키마, 프로시저 상세)**: 스키마가 자주 바뀌는 도메인은 유지 비용이 높다.
  핵심 테이블과 중요 프로시저만 유지하고, 나머지는 필요할 때 다시 분석하는 게 현실적이다.

## 습관화 실패를 막는 체크리스트

시스템이 잘 돌아가고 있는지 자가 진단:

- [ ] 매일 아침 "오늘 할일 요약해줘"를 쓰고 있는가?
- [ ] 태스크 완료 시 AI 비서에게 알려주고 있는가?
- [ ] 코드 수정 시 도메인 분석 내용을 memory에 저장하고 있는가?
- [ ] 주간보고에 AI 초안을 활용하고 있는가?
- [ ] 월 1회 메모리 정리를 하고 있는가?

5개 중 3개 이상 "예"면 잘 되고 있는 것이다.
2개 이하면 시스템이 무너지기 직전이니 가장 쉬운 것 하나만 다시 시작한다.
