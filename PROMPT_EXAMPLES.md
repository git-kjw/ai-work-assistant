## 도메인 분석

### 전체 흐름 분석

```
결제 도메인 전체 흐름 분석해줘.
비즈니스 흐름, 상태 전이, DB 스키마, 프로시저 포함해서 정리하고 memory에 저장해줘.
```

```
회원 가입부터 탈퇴까지 전체 프로세스 분석해줘.
어떤 서비스들이 관여하는지, 상태가 어떻게 바뀌는지 위주로.
```

```
채용공고 등록 도메인 분석해줘.
등록부터 노출까지 전체 흐름이랑 승인 상태 전이 포함해서.
```

### DB 스키마 / 프로시저

```
결제 도메인에서 핵심 테이블이랑 관계 정리해줘
```

```
환불 처리에 관여하는 프로시저 목록 보여줘
```

```
SP_PROCESS_REFUND 프로시저가 뭘 하는 건지 정리해줘.
비즈니스 로직이 프로시저에 있는지 Java에 있는지도 명시해줘.
```

### 로직 소재지 확인

```
환불 로직 수정해야 하는데 어디를 고쳐야 해?
Java 코드야 프로시저야?
```

```
결제 도메인에서 로직 소재지 테이블 보여줘.
어떤 기능이 Java에 있고 어떤 게 프로시저에 있는지.
```

```
야간 정산 배치 흐름 분석해줘.
어떤 프로시저를 호출하고, 어떤 테이블을 읽고 쓰는지.
```

### 상태 전이

```
주문 상태가 어떻게 바뀌는지 정리해줘.
각 전이 조건이랑 처리 주체도 포함해서.
```

```
결제 도메인에서 FAILED 상태가 되는 조건이 뭐야?
```

### 레거시 / 신규 구현 비교

```
결제 도메인에서 레거시 PC와 새 MSA API의 차이 정리해줘
```

```

```

## 아이디어

### 기록

```
아이디어 하나 기록해줘.
payment-api에서 결제 실패하면 다른 결제수단을 자동으로 제안하면 어떨까?
사용자가 "다시 시도"만 누르다가 이탈하는 거 줄일 수 있을 것 같아.
```

```
생각난 건데, member-service 회원가입 할 때 약관 동의를 한 화면에 다 보여주지 말고
단계별로 나누면 전환율이 올라가지 않을까?
```

```
엉뚱한 생각인데 기록해줘.
batch-system에서 야간 배치 실패하면 Slack 알림만 오잖아.
자동으로 1회 재시도하고, 그래도 실패하면 알림 오는 게 낫지 않을까?
```

```
이건 UX 아이디어인데, 채용공고 등록할 때 미리보기를 실시간으로 보여주면
수정 횟수가 줄어들 것 같아.
```

### 조회

```
payment-api 관련 아이디어 있었던 거 보여줘
```

```
지난달에 UX 관련해서 떠올린 거 뭐 있었지?
```

```
parked 상태인 아이디어 전체 보여줘
```

```
아직 raw 상태로 남아있는 아이디어 있어?
```

### 상태 변경

```
결제 실패 시 대안 결제수단 제안 아이디어, 팀장님한테 제안했어. proposed로 바꿔줘.
```

```
약관 동의 단계별 분리 아이디어 채택됐어. 태스크로 만들어줘.
```

```
배치 자동 재시도 아이디어는 일단 보류. parked로 바꾸고 이유는 현재 인프라에서 구현 어려움.
```
